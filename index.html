VeeCode Claude Gateway live

Corporate gateway for Claude Code. By default, your existing Anthropic Team-plan login rides through to https://claude-gateway.vee.codes for attributable spend + audit logs.

Prerequisites

  • Claude Code already installed (npm i -g @anthropic-ai/claude-code) and an Anthropic Team-plan login (claude login once) — the installer locates the binary on PATH, it does not install it for you
  • For the API-key opt-in path only: a Google Workspace account on a permitted domain and gcloud authenticated (gcloud auth login)

Install (macOS / Linux)

curl -fsSL https://veecode-claude-gateway.github.io/install.sh | bash

Install (Windows PowerShell 7+)

iwr https://veecode-claude-gateway.github.io/install.ps1 | iex

Modes

The wrapper picks one auth source per session. Default is OAuth pass-through (cases 2/3 below); set CLAUDE_GATEWAY_USE_API_KEY=1 to fall back to the gcloud → virtual-key flow (case 4).

Identity attribution

Set CLAUDE_GATEWAY_API_USER in your shell profile to your email so server-side telemetry attributes requests to you (the wrapper sends it as x-claude-gateway-user; the gateway picks it up). If unset, the header value defaults to claude-gw-user. Trust level is laptop-set, same as client_version. Billing-grade attribution on the OAuth path comes from Anthropic's Team-plan dashboard.

What it does

Drops a thin claude wrapper ahead of the real binary on PATH. In default mode it sets ANTHROPIC_BASE_URL and execs the real binary — Claude Code uses its normal logged-in OAuth token, the gateway forwards it upstream, usage bills against your Team-plan seat. With CLAUDE_GATEWAY_USE_API_KEY=1 it falls back to the gcloud → /issue-key → cached virtual key flow (case 4).

The gateway logs metadata only — timestamp, model, token counts, identity. Conversation contents are not logged.

Verify

exec $SHELL    # or open a new terminal so PATH picks up the wrapper
claude --version

Troubleshooting

"Not logged in" on claude -p

Either your Claude Code OAuth session has expired (run claude login) or you're on an older wrapper. Force a self-update:

rm -f ~/.config/claude-gateway/last_check && claude --version

"gcloud not found" / "print-access-token failed"

Only relevant on the CLAUDE_GATEWAY_USE_API_KEY=1 path. Install the gcloud CLI and run gcloud auth login.

"could not reach gateway"

Check VPN / network. The gateway is at https://claude-gateway.vee.codes.

Auth errors after working previously (API-key path)

The cached virtual key may have rotated. Force a refresh:

rm ~/.config/claude-gateway/key && claude --version

Switching gateways (advanced)

export CLAUDE_GATEWAY_URL=https://your-other-gateway.example
curl -fsSL https://veecode-claude-gateway.github.io/install.sh | bash

For repo owners: CI virtual keys

Need Claude Code to run from GitHub Actions? Mint a long-lived virtual key scoped to a single repo, with spend bucketed to the repo (not to your personal budget). Install the companion CLI: 

curl -fsSL https://veecode-claude-gateway.github.io/install-claude-gateway.sh | bash

Then, from a terminal where gcloud is logged in:

claude-gateway issue-ci-key --repo owner/name --budget 25 --ttl 90d

The CLI prints the key once — paste it into secrets.ANTHROPIC_API_KEY on the repo and set vars.ANTHROPIC_BASE_URL to the gateway URL above. Other subcommands: rotate-ci-key, revoke-ci-key, ci-key-status. Full walkthrough: docs/ci-keys.md.
